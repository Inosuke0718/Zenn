React で開発していると、「 `useState` だけで十分？」「 Jotai や Zustand みたいなライブラリは入れるべき？」「最近よく聞く React Compiler って何がすごいの？」と迷う瞬間、ありませんか？

React の状態管理は、単に「便利なライブラリを探す話」ではなく、「 **再レンダリング** 」「 **テアリング** 」「 **React の新機能との相性** 」をどう考えるかの話でもあります。

この記事では、初心者向けにこの 3 つを順番に整理しつつ、最後に「じゃあ実務ではどう選ぶのがよいのか」までまとめます。特に今は **React Compiler** が自動メモ化を担うようになってきたので、昔と同じ基準でライブラリを選ぶと設計がズレやすくなっています。

## 📍 まず結論

先に結論を書くと、 **再レンダリング最適化だけ** を目的に状態管理ライブラリを入れる必要性は、 React Compiler の登場でかなり下がりました。

いま重要なのは、「その状態は本当にクライアントで持つべきか」「外部ストアを使うなら並行レンダリング時の整合性をどう考えるか」「サーバー状態は TanStack Query のような専用ツールに分けるべきか」を切り分けることです。

初心者のうちは、まず `useState` と `useReducer` を中心に考えるのがおすすめです。そこから、クライアント全体で共有したい状態が増えたら Jotai のような細かく分割しやすい選択肢を検討し、 API 通信のキャッシュや再取得は TanStack Query に任せる、という分け方がかなりわかりやすいです。

![alt text](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/react-state-management1.png)

この図のポイントは、「ライブラリを入れるかどうか」を最初に決めるのではなく、「 **どの種類の状態か** 」を最初に分けることです。 TanStack Query はサーバー状態向け、 Jotai は細かいクライアント状態向けという整理にすると、選定がかなり楽になります。

## 🤔 なぜReactの状態管理はややこしいのか

React 初心者が最初につまずきやすいのは、状態が変わったときに「 **どこまで再描画されるのか** 」が見えにくいことです。標準的な書き方では、親コンポーネントの状態更新が子や孫の再レンダリングにつながりやすく、これがパフォーマンスの悩みとして現れます。

そのため、これまでは `React.memo` 、 `useMemo` 、 `useCallback` で手動チューニングすることがよくありました。ですが、これらは便利な反面、「どこに入れるべきか」「本当に効いているのか」を判断するのが難しく、コードも読みにくくなりがちでした。

一方で、状態管理ライブラリは「 **必要なコンポーネントだけ更新する** 」方向に設計されていることが多いです。これは大きな利点ですが、 React の外側にあるストアを読む設計にすると、 React 18 以降の **並行レンダリング** で別の難しさが出てきます。

![alt text](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/react-state-management2.png)

この図の左側が、初心者が最初に体験しやすい「広めに再レンダリングされる世界」です。右側はライブラリが得意な「ピンポイント更新」ですが、こちらには次に説明する **テアリング** の観点が入ってきます。

## 🌊 テアリング（Tearing）とは何か

React 18 では **並行レンダリング（Concurrent Rendering）** が導入され、描画処理の途中で一時停止し、優先度の高い作業を先に処理してから再開できるようになりました。

これはユーザー体験をよくする仕組みですが、描画の途中で外部ストアの値が変わると、同じ画面の中で **古い値と新しい値が混ざる** ことがあります。

この「画面が途中で裂けたように見える不整合」が、 **テアリング** です。 React は外部ストア購読のために `useSyncExternalStore` を用意しており、並行レンダリング中でも整合したスナップショットを読みやすくする目的があります。

ここで大事なのは、「ライブラリが悪い」のではなく、並行レンダリングでは「 **React の外にある値** 」をどう安全に扱うかが難しいという点です。実際、外部ストアの実装によっては、テアリング回避と新機能との相性の両立が難しいことがテストや検証でも示されています。

## 🤖 React Compilerで何が変わったか

**React Compiler** は、コンポーネントをビルド時に解析して、安全な範囲で **自動メモ化** を適用する仕組みです。

これにより、これまで手動で書いていた `React.memo` や `useMemo` や `useCallback` の一部を、開発者が意識しなくても最適化できるようになりました。

React Compiler v1.0 は 2025 年 10 月に公開され、未変更の props や子ツリーをスキップするような最適化をコンパイル段階で差し込めるようになっています。しかも、この最適化は単に `useMemo` を機械的に足すのではなく、依存関係チェックやキャッシュをより低コストな形で埋め込むよう設計されています。

![alt text](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/react-state-management3.png)

これが意味するのは、「 **再レンダリングが心配だから外部ライブラリを入れる** 」という判断が、以前ほど強い理由ではなくなったことです。これからは、ライブラリを入れる理由を「再レンダリング対策」ではなく、「 **状態の分割しやすさ** 」「 **派生状態の書きやすさ** 」「 **チームでの保守性** 」に置き換えて考える方が自然です。

## 🛠️ じゃあ実務でどう選ぶ？

初心者向けにかなり単純化すると、選び方は次の表で整理できます。

| 状況 | まず選ぶもの |
|---|---|
| 画面内だけで閉じる状態 | `useState` / `useReducer` |
| API 通信、キャッシュ、再取得 | ** TanStack Query ** |
| 細かい依存関係や派生状態が多い | ** Jotai ** などのアプローチ |
| 「とにかく再レンダリングが不安」 | まず ** React Compiler ** 前提で見直す |

特に **サーバー状態** と **クライアント状態** を混ぜないのは重要です。 TanStack Query はサーバー状態向けとして扱うのが定石で、クライアント側の UI 状態とは役割を分けるのがよいとされています。

また、 Jotai のようなアプローチは、状態を細かく分割したいときや派生状態が多いときに相性がよいです。逆に、単に「再レンダリングを減らしたいだけ」なら、 React Compiler が効く前提で標準機能から始めた方が、学習コストも設計コストも低く抑えやすいです。

## 📖 まとめ

昔は「再レンダリング対策のために状態管理ライブラリを入れる」が強い動機でしたが、今は「 **その状態の性質に合った道具を選ぶ** 」が中心になってきました。

React Compiler の登場で、 React 標準機能はかなり戦いやすくなっているので、初心者ほどまずは標準機能を軸に学ぶ価値があります。

- ** 再レンダリング対策は React Compiler に任せる ** 時代へ
- ** サーバー状態（API）は TanStack Query ** で管理するのがベストプラクティス
- クライアント状態は ** まずは useState / useReducer ** から。複雑になったらライブラリを検討
- ライブラリ選びはパフォーマンスよりも ** 「設計のしやすさ」 ** を重視する

まずは公式ドキュメントで ** React Compiler ** の仕組みをチェックしてみましょう！ 効率的な設計で、楽しい React ライフを送ってくださいね。
