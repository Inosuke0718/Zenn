## はじめに

最近、AI開発まわりで **Loop Engineering** という言葉を見かけるようになってきました。

これまでのAI活用では、

> いかに良いプロンプトを書くか

が重要でした。

しかし、Claude CodeやCodexのようなAIエージェントを使う場面では、少し考え方が変わります。

重要になるのは、毎回人間が細かく指示することではありません。

> AIが自分で作業し、検証し、必要なら修正し、完了条件に到達するまで回り続ける「ループ」をどう設計するか

です。

Claude Agent SDKの公式ドキュメントでも、エージェントは「プロンプトを受け取る → Claudeが判断する → ツールを実行する → 結果を受け取って再判断する → 完了したら結果を返す」というループ構造で動くと説明されています。

つまり、Loop Engineeringとはざっくり言えば、

> AIに一回だけ命令する技術ではなく、AIが何度も判断・実行・検証できる仕組みを設計する技術

です。

この記事では、Loop Engineeringを次の観点で整理します。

* 4つのループ設計
* ループの質を上げる設計原則
* 評価モデル・停止条件・コスト管理
* 実務でハマりやすい罠
* AI開発における本質的な使い分け

---

## まず「Agent Loop」とは何か

Claude CodeのAgent Loopは、かなりシンプルに言えば次の流れです。

```text
ユーザーの指示
↓
AIが状況を判断する
↓
必要なツールを呼び出す
↓
ツールの結果を受け取る
↓
もう一度AIが判断する
↓
完了するまで繰り返す
```

たとえば、ユーザーが

```text
auth.ts のテストが落ちているので直して
```

と頼んだとします。

このときAIは、いきなり修正するのではなく、次のようなループを回します。

```text
1. npm test を実行する
2. エラー内容を読む
3. 関連ファイルを読む
4. 原因を推測する
5. コードを編集する
6. もう一度テストを実行する
7. テストが通れば完了する
8. 通らなければ再度調査する
```

公式ドキュメントでは、この1回分の「AIの判断 → ツール実行 → 結果を戻す」流れをターンとして扱い、`max_turns` / `maxTurns` で上限を設定できると説明されています。さらに、`max_budget_usd` / `maxBudgetUsd` によってコスト上限も設定できます。

ここが重要です。

AIエージェントの本体は、単なる「チャット」ではありません。

> 判断と実行を繰り返すループ

です。

そしてLoop Engineeringでは、このループが暴走せず、目的に向かって収束するように設計します。

---

## Loop Engineeringは「プロンプトを書くこと」ではない

プロンプトエンジニアリングでは、主に1回の入力文を工夫します。

たとえば、

```text
あなたは優秀なエンジニアです。
以下のコードをレビューしてください。
出力は問題点・改善案・修正コードの順にしてください。
```

のような指示です。

これは今でも重要です。

ただ、AIエージェント開発ではそれだけでは足りません。

なぜなら、エージェントは1回で終わるとは限らないからです。

コードを読む。
テストを実行する。
修正する。
またテストする。
エラーが出たら別の原因を探す。
必要ならユーザーに確認する。

このように、AIは何度も行動します。

そのため、設計対象は「1つのプロンプト」から「ループ全体」へ広がります。

Loop Engineeringで設計するべきものは、たとえば次のような項目です。

```text
- 何をゴールにするか
- 何を成功条件にするか
- どのツールを使わせるか
- どのタイミングで停止するか
- 失敗したらどうするか
- 誰が評価するか
- 人間に確認すべき条件は何か
- コスト上限をどうするか
```

つまり、Loop Engineeringとは、

> AIに作業させることではなく、AIが安全に作業できる環境を設計すること

だと言えます。

---

## 1. 4つのループ設計

Loop Engineeringは、人間の関与度によって大きく4つに分けて考えると理解しやすいです。

```text
手動寄り
↓
① ターン駆動ループ
② ゴール駆動ループ
③ 時間駆動ループ
④ 自律駆動ループ
↓
自動寄り
```

それぞれ見ていきます。

---

## ① ターン駆動ループ

ターン駆動ループは、もっとも基本的なループです。

ユーザーの1つの指示を起点に、AIが

```text
作業する
↓
確認する
↓
必要なら修正する
↓
完了する
```

という流れを回します。

![ターン駆動ループ](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/loop-engineering/turn-based-loop.png)

たとえば、

```text
このReactコンポーネントを改善して。
ただし、修正後にnpm testを実行し、テストが通ったことを確認して。
```

のような指示です。

この場合、AIは単にコードを書くだけでなく、

```text
1. 現在のコードを読む
2. 修正する
3. テストを実行する
4. 結果を確認する
5. 必要なら再修正する
```

というループを回します。

### ターン駆動ループの停止条件

ターン駆動ループの停止条件は主に2つです。

```text
- タスクが完了した
- 追加の情報が必要になった
```

たとえば、AIが

```text
この仕様だとA案とB案のどちらで実装すべきか判断できません
```

と返してきたら、人間の追加コンテキストが必要な状態です。

ここで無理に進ませると、AIは勝手な前提で実装し始めます。

### 実践のコツ

ターン駆動ループでは、AIに必ず「自己検証」をさせることが重要です。

悪い指示はこれです。

```text
いい感じに直して
```

良い指示はこれです。

```text
以下の条件を満たすように修正して。
- TypeScriptの型エラーが出ない
- npm test が通る
- 既存のUIを崩さない
- 不要な依存を追加しない
```

AIは「いい感じ」の判断が苦手です。

逆に、

```text
テストが通る
型エラーがない
指定されたファイルだけ変更する
```

のようにチェック項目が具体的だと、自己検証しやすくなります。

---

## ② ゴール駆動ループ

ゴール駆動ループは、最終目標をAIに渡し、その達成まで試行錯誤させるループです。

![ゴール駆動ループ](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/loop-engineering/goal-based-loop.png)

たとえば、

```text
ログイン画面を完成させて。
要件は以下。
- Googleログインができる
- エラー時にメッセージを表示する
- ログイン後は /home に遷移する
- Playwrightで主要動線を確認する
```

のように、ゴールと受け入れ条件をセットで渡します。

このときAIは、細かい手順を人間に毎回聞くのではなく、自分で必要な作業を分解します。

```text
1. 認証まわりのコードを読む
2. Supabase設定を確認する
3. UIを修正する
4. callback処理を確認する
5. 動作テストする
6. 問題があれば修正する
```

### ゴール駆動ループの停止条件

ゴール駆動ループでは、停止条件を必ず明示する必要があります。

代表的には次の2つです。

```text
- ゴールが達成された
- 最大ターン数に到達した
```

Claude Agent SDKでも、ループの暴走を防ぐために `max_turns` / `maxTurns` や `max_budget_usd` / `maxBudgetUsd` を設定でき、上限に達した場合は `error_max_turns` や `error_max_budget_usd` のような終了状態が返ると説明されています。

つまり、実務では

```text
完了するまで頑張って
```

ではなく、

```text
最大20ターンまで試行して。
解決できなければ、原因・試したこと・次に必要な情報をまとめて停止して。
```

のように設計するべきです。

### 超重要：評価専用モデルを分ける

ゴール駆動ループで特に重要なのが、評価の分離です。

AIに作業させたあと、同じAIに

```text
自分の実装は正しいですか？
```

と聞くと、評価が甘くなりがちです。

人間でも、自分が作ったものを自分でレビューすると見落としが増えます。

AIも同じです。

そこで有効なのが、

> 作業するAIと評価するAIを分ける

という設計です。

たとえば、

```text
実装エージェント：
- コードを読む
- 修正する
- テストする

評価エージェント：
- 仕様と照らし合わせる
- 変更差分をレビューする
- バグや仕様漏れを指摘する
- 点数をつける
```

のようにします。

Claude Agent SDKにはサブエージェントの仕組みがあり、サブエージェントは親の会話履歴を引き継がず、新しい会話として開始されるため、メインエージェントの長い文脈をそのまま抱え込まずにタスクを分離できます。

この「文脈が新鮮な別エージェントに評価させる」という考え方は、AI開発ではかなり重要です。

いわゆる **LLM Judge** の設計です。

## ③ 時間駆動ループ

時間駆動ループは、時間をトリガーにしてAIを動かすループです。

たとえば、

```text
- 毎朝9時にIssueを確認する
- 1時間ごとにエラーログを確認する
- 毎週月曜に依存パッケージの更新を確認する
- 30分後に再チェックする
```

のようなものです。

これは、AIの内側というより、AIを起動する外側の仕組みです。

イメージとしては、

```text
cron
↓
AIエージェント起動
↓
状態確認
↓
必要なら作業
↓
結果を通知
```

という流れです。

### 時間駆動ループの停止条件

時間駆動ループでは、停止条件は少し特殊です。

```text
- 予定された実行が終わる
- 人間がキャンセルする
- エラーや予算上限で止まる
```

たとえばローカルPC上で動かしている場合、PCの電源が切れれば当然止まります。

本格的に運用するなら、ローカルではなくクラウド上で実行する方が安定します。

```text
- GitHub Actions
- Cloud Run
- Lambda
- Supabase Edge Functions
- Cloudflare Workers
- cron
```

などが候補になります。

### 実践のコツ

時間駆動ループでは、毎回LLMをフル稼働させるとコストが膨らみます。

そのため、

```text
まずスクリプトで状態確認する
↓
異常があるときだけLLMを呼ぶ
```

という設計が重要です。

たとえば、エラーログ監視なら、

```text
1. スクリプトで直近のエラー件数を確認
2. エラーがしきい値を超えたらAIに分析させる
3. 修正が必要ならIssue化する
```

のようにします。

毎回AIに全部読ませる必要はありません。

## ④ 自律駆動ループ

自律駆動ループは、もっとも自動化レベルが高いループです。

常にバックグラウンドで走り続け、

```text
- エラー検知
- 依存関係の更新
- セキュリティ警告の確認
- PRの自動レビュー
- テスト失敗のトリアージ
- ドキュメントの更新漏れ検知
```

などを行います。

これは、もはや「AIに作業を頼む」というより、

> AIを含んだ小さな運用システムを作る

に近いです。

![プロアクティブループ](https://raw.githubusercontent.com/Inosuke0718/Zenn/main/images/loop-engineering/proactive-loop.png)

### 自律駆動ループの停止条件

自律駆動ループの停止条件は、基本的には人間の明示的な停止です。

```text
- システムをOFFにする
- 監視対象を外す
- 予算上限に達する
- 危険な操作を検知して停止する
```

ここで重要なのは、自律駆動ループほど安全設計が必要になるということです。

Claude Agent SDKの権限設定では、ツールを自動承認する `allowed_tools` / `allowedTools`、ブロックする `disallowed_tools` / `disallowedTools`、承認モードを制御する `permission_mode` / `permissionMode` などを使って、エージェントが実行できる操作を制御できます。

自律駆動ループでは、AIに何でも許可してはいけません。

特に、

```text
- 本番DBへの書き込み
- rm -rf 系の危険コマンド
- 認証情報へのアクセス
- 外部送信
- 課金が発生する操作
```

は厳しく制限する必要があります。

---

## ループは入れ子にできる

Loop Engineeringで面白いのは、ループを入れ子にできることです。

たとえば、大きな自律ループの中に、小さなゴール駆動ループを入れられます。

```text
マクロループ：
毎朝、プロジェクト全体の状態を確認する

↓ 必要に応じて起動

ミクロループA：
テスト失敗を修正する

ミクロループB：
依存パッケージ更新の影響を調査する

ミクロループC：
ドキュメントの差分を更新する
```

このように、

```text
大きな監視ループ
↓
必要なときだけ小さな修正ループを起動
↓
結果を評価
↓
次の判断に反映
```

という構造にできます。

これができると、AIエージェントは単なるチャット相手ではなく、開発ワークフローの一部になります。

## 2. ループの質を上げる4つの設計原則

Loop Engineeringで大事なのは、AIの性能だけではありません。

AIを取り巻く環境の設計が重要です。

ここでは、ループの質を上げる4つの原則を整理します。

---

## 原則1：AIに検証する手段を与える

AIにコードを書かせるだけでは不十分です。

重要なのは、

> AI自身が、自分の作業結果を確認できる状態にすること

です。

たとえばフロントエンド開発なら、AIに次のような手段を与えると精度が上がります。

```text
- テストを実行できる
- ブラウザで表示を確認できる
- スクリーンショットを撮れる
- console error を読める
- lintを実行できる
- 型チェックできる
```

Claude Agent SDKでは、ファイル操作の `Read` / `Edit` / `Write`、検索の `Glob` / `Grep`、コマンド実行の `Bash`、Web関連の `WebSearch` / `WebFetch` などの組み込みツールが用意されています。

つまり、AIに単に

```text
修正して
```

と言うのではなく、

```text
修正後に npm test と npm run typecheck を実行し、
失敗した場合は原因を分析して再修正してください。
```

と指示するべきです。

AIに「目」と「手」と「確認手段」を与える。

これがループの品質を大きく左右します。

---

## 原則2：コードレビューには第2のエージェントを使う

AIが自分で書いたコードを、自分でレビューすると甘くなります。

これはかなり重要です。

なぜなら、同じ会話文脈の中にいるAIは、自分の判断を前提にしがちだからです。

そこで有効なのが、

```text
実装エージェント
↓
文脈を切ったレビューエージェント
↓
必要なら修正ループに戻す
```

という設計です。

レビュー用エージェントには、実装の過程を細かく渡しすぎない方がよい場合もあります。

渡すべきなのは、たとえば次の情報です。

```text
- 元の要件
- 変更差分
- テスト結果
- 受け入れ条件
```

逆に、

```text
- 実装エージェントの言い訳
- 途中の推測
- 自己評価
```

は不要なことがあります。

レビューAIには、なるべくフレッシュな状態で見てもらう。

これが **Context Fresh** の考え方です。

---

## 原則3：コードベースを綺麗にする

AIエージェントの性能は、コードベースの状態に強く影響されます。

人間にとって読みにくいコードは、AIにとっても読みにくいです。

たとえば、

```text
- ファイル名がバラバラ
- 同じ処理が複数箇所にある
- 型が曖昧
- 不要なコメントが多い
- READMEが古い
- ディレクトリ構成に一貫性がない
```

このような状態だと、AIは迷います。

迷ったAIは、余計なファイルを読みます。
余計なファイルを読むと、コンテキストを消費します。
コンテキストが膨らむと、コストも上がり、判断もブレます。

Claude Agent SDKのドキュメントでも、会話履歴・ツール入力・ツール出力・大きなファイルの読み込みなどがコンテキストを消費し、長いセッションほどコンテキストが大きくなると説明されています。

つまり、コードベースを綺麗にすることは、AI時代においてさらに重要になります。

---

## 原則4：ドキュメントに届きやすくする

AIは、必要な情報にアクセスできなければ推測します。

そして、推測が増えるほどハルシネーションも増えます。

そのため、AIが迷わず仕様やルールに到達できる導線を作ることが重要です。

たとえば、

```text
- README.md
- CLAUDE.md
- docs/spec.md
- docs/architecture.md
- docs/api.md
- docs/design-system.md
```

のようなドキュメントを整備します。

特にClaude Codeでは、プロジェクトのルールや永続的な指示は初回プロンプトだけに書くのではなく、`CLAUDE.md` のようなプロジェクト文脈に置くことが重要です。公式ドキュメントでも、コンテキスト圧縮によって初期の具体的な指示が保持されない可能性があるため、永続ルールは `CLAUDE.md` に置くべきだと説明されています。

AIにとっての良いドキュメントとは、長いドキュメントではありません。

```text
- どこに何があるか分かる
- 判断基準が明確
- 禁止事項が書かれている
- 受け入れ条件が書かれている
- 更新されている
```

このようなドキュメントです。

## 3. 運用上の罠とコスト管理

Loop Engineeringで一番怖いのは、AIが迷走することです。

ループは便利ですが、設計を間違えると危険です。

```text
- いつまでも終わらない
- 同じ失敗を繰り返す
- 余計なファイルを読み続ける
- 不要な修正を広げる
- トークンを燃やし続ける
- 間違った成功判定で終わる
```

こうした問題を防ぐには、泥臭い設計が必要です。

---

## 罠1：成功条件が曖昧

一番よくないのは、

```text
いい感じに改善して
```

です。

これだとAIは、いつ終わればいいのか判断できません。

良いループには、必ず成功条件があります。

```text
成功条件：
- npm test が通る
- npm run typecheck が通る
- ログイン後に /home へ遷移する
- エラー時にユーザー向けメッセージが表示される
- 変更ファイルは auth 関連に限定する
```

成功条件が明確であれば、AIはそこに向かって収束できます。

---

## 罠2：停止条件がない

Loop Engineeringでは、停止条件が超重要です。

```text
- 最大ターン数
- 最大コスト
- 最大実行時間
- 最大リトライ回数
- 人間確認に切り替える条件
```

を決めておく必要があります。

たとえば、

```text
最大10ターンまで試してください。
10ターン以内に解決できない場合は、以下をまとめて停止してください。

- 試したこと
- 分かったこと
- 解決できていない原因
- 人間が確認すべきポイント
```

のようにします。

公式ドキュメントでも、上限なしの場合はClaudeが自分で完了するまでループが続くため、プロダクション用途では予算設定が良いデフォルトになると説明されています。

---

## 罠3：決定論的な作業にLLMを使ってしまう

これはかなり本質的です。

LLMは便利ですが、すべてをLLMにやらせるべきではありません。

特に、

```text
- 必ず同じ結果になる処理
- 単純なファイル変換
- 定型的な置換
- JSONの整形
- CSVの集計
- ログ件数のカウント
```

のような作業は、普通のスクリプトでやるべきです。

悪い例：

```text
AIに毎回ログを全部読ませて、エラー件数を数えさせる
```

良い例：

```text
スクリプトでエラー件数を集計する
しきい値を超えたときだけAIに原因分析させる
```

LLMは「判断が必要なところ」に使うべきです。

逆に、毎回同じ結果になる処理はスクリプトで十分です。

ここを間違えると、コストも遅延も一気に増えます。

## 罠4：コンテキストを肥大化させる

AIエージェントは、作業中にどんどん情報を読み込みます。

```text
- ファイルの中身
- テスト結果
- エラーログ
- 会話履歴
- ツールの実行結果
- ドキュメント
```

これらはすべてコンテキストを消費します。

Claude Agent SDKでは、長時間のエージェント運用において、サブエージェントの活用、ツールの絞り込み、MCPツールスキーマのコスト管理、低いeffort設定の活用などがコンテキスト効率化の手段として挙げられています。

実務では、AIに何でも読ませるのではなく、

```text
- まず関連ファイルを検索する
- 必要な範囲だけ読む
- 大きなログは要約して渡す
- サブタスクはサブエージェントに分ける
```

という設計が必要です。

## メタ・ハーネスという考え方

Loop Engineeringでさらに重要になるのが、メタ・ハーネスです。

これは簡単に言えば、

> AIが個別のバグを直すだけでなく、なぜそのバグが起きたのかを分析し、仕組みそのものを改善するループ

です。

たとえば、AIがテスト失敗を直したとします。

普通のループなら、ここで終わりです。

```text
テスト失敗
↓
AIが修正
↓
テスト成功
↓
完了
```

しかし、メタ・ハーネスではさらに考えます。

```text
なぜこのバグが起きたのか？
↓
同じバグを防ぐテストはあるか？
↓
ドキュメントに不足はないか？
↓
Lintルールで防げないか？
↓
CLAUDE.mdにルールを追加すべきか？
```

つまり、個別対応で終わらせず、次回以降のループの精度を上げる仕組みです。

Claude Agent SDKには、ツール実行前の `PreToolUse`、実行後の `PostToolUse`、エージェント終了時の `Stop`、コンテキスト圧縮前の `PreCompact` などのhooksがあり、ツール呼び出しの検証・ブロック・監査・保存などに使えると説明されています。

このようなhooksを使うと、

```text
AIが作業する
↓
結果を検証する
↓
ログを保存する
↓
失敗パターンを分析する
↓
次回のルールを改善する
```

という改善ループを作れます。

これがLoop Engineeringのかなり重要な方向性だと思います。

## 実務で使えるループ設計テンプレート

最後に、実務で使いやすいテンプレートを置いておきます。

AIエージェントに依頼するときは、次のように書くとループが安定します。

```text
## Goal
何を達成したいかを書く

## Context
前提情報を書く

## Scope
触ってよい範囲を書く

## Tools
使ってよいツールを書く

## Acceptance Criteria
成功条件を書く

## Verification
検証方法を書く

## Stop Conditions
停止条件を書く

## Escalation
人間に確認すべき条件を書く
```

具体例です。

```text
## Goal
SupabaseのGoogleログインが成功するように修正してください。

## Context
Next.js App Routerを使っています。
ログイン後は /home に遷移します。

## Scope
変更してよい範囲は auth 関連ファイルのみです。
UI全体のデザイン変更はしないでください。

## Tools
ファイル読み取り、編集、npm test、typecheckは実行して構いません。

## Acceptance Criteria
- Googleログインボタンを押すと認証フローに進む
- callback後に /home に遷移する
- エラー時に画面上に分かりやすいメッセージが出る
- TypeScriptエラーがない

## Verification
- npm run typecheck を実行する
- npm test があれば実行する
- 変更差分をレビューする

## Stop Conditions
最大10ターンまで。
解決できない場合は、試したこと・原因候補・人間が確認すべき設定をまとめて停止してください。

## Escalation
Supabase側のProvider設定やGoogle Cloud Consoleの設定が必要な場合は、勝手に推測せず人間に確認してください。
```

このように書くと、AIはかなり動きやすくなります。

## まとめ

Loop Engineeringは、単なる流行語ではありません。

AIエージェント時代の開発では、重要な設計対象が変わっています。

これまでは、

```text
良いプロンプトを書く
```

ことが中心でした。

しかしこれからは、

```text
AIが判断・実行・検証・停止できるループを設計する
```

ことが重要になります。

特に大事なのは次のポイントです。

```text
- ゴールを明確にする
- 成功条件を定量化する
- 停止条件を決める
- AIに検証手段を与える
- レビューは別エージェントに任せる
- 決定論的作業はスクリプトに逃がす
- コンテキストとコストを管理する
- 失敗からルール自体を改善する
```

Loop Engineeringの本質は、AIを放置することではありません。

むしろ逆です。

> AIが安全に自律できるように、環境・評価・停止条件・権限・検証手段を人間が設計すること

です。

AIに全部任せるのではなく、AIが正しく回る仕組みを作る。

これが、これからのAI開発におけるかなり重要なスキルになっていくと思います。
