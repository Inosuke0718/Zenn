# 【OpenAI公式ベース】Prompt Engineering実践ポイントと、モダンUI設計ガイド

## はじめに

AIに高品質なUIや実用的なコードを生成させるには、単に「何を作るか」を伝えるだけでなく、 **どう指示するか** と、 **どんな制約を与えるか** の両方を設計する必要があります. 
構造化、再利用、例示、関連コンテキストの付与、モデル特性に応じた使い分けが重視されており、フロントエンド設計では制約、視覚参照、ページ構成、検証の重要性が繰り返し示されています.
また、Apps SDKのUI guidelinesでは、ChatGPT内で動くアプリ向けに、システムカラーやシステムフォントを尊重した、ネイティブ感のあるUI設計が推奨されています.

この記事では、OpenAI公式の内容をもとに、エンジニアやデザイナーが実務で使いやすい形に整理して紹介します.

## 1. Prompt engineeringの実践ポイント

OpenAIの公式Prompt engineeringガイドを、実務向けに整理すると、次の観点が重要です.

### 1-1. 役割と出力形式を最初に固定する

OpenAI公式では、system / developer / user などのmessage rolesを使い分けながら、モデルに期待する振る舞いを明示する考え方が紹介されています.
そのため、「何をしてほしいか」だけでなく、「どんな立場で」「どの形式で」出力するかまで先に固定すると、回答のブレを抑えやすくなります.

例:

- あなたはSaaSのUI設計に強いフロントエンドエンジニアです.
- React + Tailwindで実装可能な形で出力してください.
- 説明のあとに、コンポーネント構成案とコード例を返してください.

### 1-2. プロンプトを構造化する

OpenAI公式では、MarkdownやXMLのような構造を使って、指示・条件・入力データ・期待出力を分けて書く方法が推奨されています.
長い自然文を一気に渡すよりも、セクションを分けたほうが、モデルが条件を読み落としにくくなります.

例:

- Goal: 何を作るか.
- Constraints: 使う技術、禁止事項、優先順位.
- Context: プロダクトの前提や対象ユーザー.
- Output format: コード、解説、チェックリストの順番.

### 1-3. Few-shotで期待値をそろえる

OpenAI公式ガイドでは、few-shot prompting、つまり「こういう入力にはこう返してほしい」という例を与える方法が紹介されています.
UI生成やコード生成でも、1つ良いサンプルを添えるだけで、命名規則、粒度、トーン、出力フォーマットが安定しやすくなります.

例:

- 期待するコンポーネント分割の例を1つ見せる.
- 望ましいTailwindの書き方を1例だけ添える.
- レビュー結果の書式を先に示す.

### 1-4. 関連コンテキストを十分に渡す

OpenAI公式では、関連するコンテキストを渡すことが、出力品質の改善に有効だと説明されています.
たとえば、既存のデザインシステム、ブランドトーン、ユースケース、参考UI、既存コードの一部を渡すと、抽象的な“いい感じ”よりも一貫した出力を得やすくなります.

渡すと有効なもの:

- デザイントークン.
- 既存コンポーネントのAPI.
- ページの目的と想定ユーザー.
- 良い参考UIのスクリーンショットやムードボード.

### 1-5. モデル特性に合わせて指示を変える

OpenAI公式では、GPT系モデルとreasoning modelでは、効果的なプロンプトの作り方が異なることが説明されています.
そのため、常に「考える過程を全部書かせる」のではなく、モデルに応じて、必要な制約や評価観点を明示するほうが実務では安全です.

実務での考え方:

- GPT系モデルには、役割・制約・出力形式を明確に渡す.
- reasoning modelには、目標、制約、評価軸をはっきり与える.
- どちらでも、曖昧な依頼より具体的な成功条件を渡す.

### 1-6. 反復と検証を前提にする

OpenAIのフロントエンド設計記事では、モデルが自分の仕事を検査・検証する能力や、ツール利用による確認の重要性が強調されています.
また、React + Tailwindを出発点にしつつ、Playwrightのようなツールで複数ビューポートや状態遷移を確認する流れが紹介されています.
つまり、良いプロンプトは“一発で完璧に出す魔法”というより、生成と検証を回しやすくする設計だと捉えるのが自然です.

## 2. モダンなフロントエンド設計の考え方（GPT-5.4のクセをねじ伏せる引き算の美学）

OpenAIの「Designing delightful frontends with GPT-5.4」では、モデルに自由に作らせるだけでは、凡庸でテンプレ感のあるUI（無駄なカードUIや過剰な装飾）に寄りやすいことが示されています.
そのため、質の高いUIを出したいなら、最初に**「やってはいけないこと（ハードルール）」**を与え、「引き算のデザイン」を徹底させることが重要です.

### 2-1. 「ハードルール」と制約を先に定義する

OpenAI公式は、デザインシステムや制約を最初に定義してから生成を始めることを勧めています. GPT-5.4の「盛りたがるクセ」を抑えるため、以下のような具体的な禁止事項・絶対ルールをプロンプト（Cursorのrulesなど）に組み込みます.

 **絶対ルール（マスト要件）の例:** 
- 最初の画面（ヒーローセクション）は1つの構図（ポスター）として成立させる.
- ロゴを隠しても自社サービスだとわかる世界観を構築する.
- 背景の無意味なベタ塗りを避け、グラデーションや高品質な写真を使用する.

 **禁止事項（やらないこと）の例:** 
- 小さな部品をごちゃごちゃ並べない（無駄な枠線やカードUIの乱用を禁止）.
- 見出しがブランド名より目立ってはならない.
- 中身のない冗長な文章（ポエム）を書かない.
- 差し色は原則1色のみ（カラフルにしすぎない）.
- 書体（フォント）は最大2つまで.

### 2-2. 視覚参照を渡す

OpenAI公式では、スクリーンショットやムードボードなどの視覚参照を与えることが推奨されています.
言葉だけで「洗練されたUI」と伝えるより、参考となる見た目の方向性を見せたほうが、レイアウト、余白感、密度、雰囲気がそろいやすくなります. 

### 2-3. ページを“物語”として設計する

OpenAI公式は、ランディングページの流れを、Hero、Supporting imagery、Product detail、Social proof、Final CTAのようなストーリーとして組む考え方を紹介しています.
これは単にセクションを並べるのではなく、「最初に何を伝え、その次に何で納得してもらい、最後に何をしてもらうか」を設計する、ということです. 
また、見出しは**PCで3行、スマホで1行**で読める長さにし、文章全体を削る**「30%削除ルール」**を適用してスッキリさせることが推奨されています.

### 2-4. 目的によるUIの使い分けと量産型UIの回避

AIにUIを生成させる際、「マーケティングUI（LPなど）」と「プロダクトUI（SaaSのダッシュボードなど）」でプロンプトの指示を明確に分ける必要があります.

- **マーケティングUI:** ブランドの世界観を重視し、製品固有の言葉でユーザーの感情を動かす見出しにする.
- **プロダクトUI:** 雰囲気よりも「状態と操作」を最優先し、見出しには「何ができるか（What can do）」を端的に記載する.

どちらの場合も、よくある“白いカードを並べただけのSaaS風UI”から抜けるには、装飾を増やすより、**構図・階層・余白を整理する（引き算をする）** ほうが効果的です. フルブリード（全幅表示）のヒーロー画像を活用したり、Framer Motion等で適切なモーション（階層や操作感を示すアニメーション）を加えることで品質が向上します.

### 2-5. 最終品質の7つのチェックリスト

生成されたUIが本当に実用的か、以下の項目でレビュー（またはAI自身にセルフチェック）させましょう.

1. 一番最初の画面で「ブランド」がひと目で伝わるか？
2. 画面内に「視覚的な主役」が存在しているか？
3. 見出しを読むだけで、ページの内容が理解できるか？
4. 各セクションに「1つの明確な目的」が与えられているか？
5. カードUIを使っている場合、それは本当に必要か？（思考停止のカード配置になっていないか？）
6. 階層や雰囲気を伝える適切なモーションが入っているか？
7. ドロップシャドウに頼りすぎて高級感を演出しようとしていないか？

## 3. ChatGPT Apps向けUIガイドとの違い

ここで注意したいのは、一般的なWebフロントエンド向けのガイドと、Apps SDKのUI guidelinesは、対象が少し異なることです.
前者はブランド表現を含む一般フロントエンド設計の話で、後者は「ChatGPTの中でネイティブに感じるアプリ体験」を作るための指針です.

Apps SDKのUI guidelinesでは、次のようなルールが明示されています.

- システムカラーを優先し、ブランド色はアクセントとして使う.
- ChatGPTのミニマルな見た目を壊すカスタムグラデーションや派手なパターンは避ける.
- フォントはシステムフォントを継承し、カスタムフォントは使わない.
- カードでは深いナビゲーションやネストしたスクロールを避ける.
- フルスクリーンでもChatGPTのcomposerと共存する前提でUXを組む.

このため、「一般SaaSのUIを作る」ときと、「ChatGPT内アプリを作る」ときでは、同じ“モダン”でも最適解が少し変わります.
一般Webではブランドの個性を強める余地がありますが、ChatGPT Appsではプラットフォームとの一貫性を優先したほうが自然です.

## 4. 実務で使うときのプロンプト例

最後に、上の内容を踏まえた、実務向けの安全なプロンプト例を置いておきます.

### UI設計用プロンプト例

あなたはSaaSの管理画面設計に強いフロントエンドエンジニアです。  
React + Tailwindで実装可能なUIを提案してください。  
目的は「情報の把握」と「主要操作の完了」を最短で行える画面です。  
以下の制約を守ってください。  

- 色はニュートラル中心で、アクセントカラーは1色のみ.
- カードの乱用は禁止し、余白と階層で情報を整理する.
- 各セクションの役割は1つに絞る.
- 先に画面構成を説明し、その後にコンポーネント案、最後にコードを出力する.
- 必要なら複数ステップに分けて提案する.

参考として渡すもの:

- 既存デザインシステム.
- 参考スクリーンショット.
- 対象ユーザーと主要ユースケース.
- 既存コンポーネントの制約.

## 5. おわりに

OpenAI公式の内容を実務に引き寄せてまとめると、重要なのは「うまいお願いの仕方」ではなく、 **構造化された指示、十分なコンテキスト、明確な制約、そして検証可能なワークフロー** です.
また、UIガイドは1つではなく、一般フロントエンド向けの原則と、ChatGPT Apps向けの原則を分けて理解すると、記事としても実装としてもブレにくくなります.
