# はじめに

「AIコーディングツール、使ってみたけど思ったより微妙だった…」

そんな経験ありませんか?実は、多くの人がAIツールを**チャットボット**として使ってしまい、本来の力を引き出せていません。

この記事では、Google Antigravityなどの最新AIコーディングツールを使って、**プロ品質のアプリを爆速で作る「FLOWフレームワーク」**を初学者向けに解説します。

## この記事で学べること

- AIに「お願い」するのではなく「指示」する方法
- 手戻りを防ぐ開発プロセスの組み立て方
- 複数のAIエージェントを同時に働かせる技術
- 実際のプロジェクトでの使い方

## FLOWフレームワークとは？

FLOWは、AI開発コンサルタントのJack Roberts氏が提唱する、4ステップの開発フレームワークです。

```
F - Frame（構想・定義）
L - Layout（設計・デザイン）
O - Orchestration（構築・指揮）
W - Workflows（自動化・保守）
```

重要なのは、**「いきなりコードを書かない」**こと。料理に例えるなら、レシピも材料も確認せずにいきなり調理を始めるようなものです。

# Step 1: Frame（構想・定義）

## なぜフレーム作りが重要なのか

AIは「曖昧な指示」に弱いです。「かっこいいサイト作って」では、AIも困ってしまいます。

### やること

1. **ClaudeやChatGPT、Geminiと対話する**
   - Antigravityではなく、Claude、ChatGPT、Geminiなどの対話型AIを使います
   - 理由: じっくり考えを整理するには、会話型AIが適しているから

2. **アイデアを洗練させる**

   ```
   プロンプト例:
   「私はToDoアプリを作りたいと思っています。
   この考えに異議を唱えて（challenge my thinking）、
   より良いアイデアにしてください」
   ```

3. **SOP（標準作業手順書）を作成させる**
   - 最終目標: **500文字以内**の明確な指示書
   - これをAntigravityに渡します

### 実践例

❌ 悪い例:

```
「タスク管理アプリを作って」
```

✅ 良い例:

```
# タスク管理アプリ仕様書

## 目的
個人開発者向けの、シンプルなタスク管理アプリ

## 機能要件
- タスクの追加・編集・削除
- カテゴリー別の色分け
- 期限の視覚化（残り日数表示）
- ダークモード対応

## 技術スタック
- フロントエンド: React + TypeScript
- スタイリング: Tailwind CSS
- データ保存: LocalStorage

## デザイン方針
- ミニマルで直感的なUI
- モバイルファースト
```

:::message
**ポイント**: 「何を作るか」だけでなく「なぜ作るか」「誰が使うか」まで明確にしましょう
:::

# Step 2: Layout（設計・デザイン）

## デザインルールを「ファイル化」する

ここでは、AIに参照させるデザインガイドを用意します。

### 1. スタイルガイドファイルの作成

`design-system.md` のような名前でファイルを作成:

```markdown
# デザインシステム

## カラーパレット

- Primary: #3B82F6（青）
- Secondary: #10B981（緑）
- Background: #F9FAFB（明るいグレー）
- Text: #1F2937（ダークグレー）

## タイポグラフィ

- フォント: Inter（Googleフォント）
- 見出し: 32px / Bold
- 本文: 16px / Regular
- キャプション: 14px / Light

## スペーシング

- セクション間: 48px
- 要素間: 24px
- 内部余白: 16px

## コンポーネントスタイル

- ボタン: 丸角8px、影なし、ホバー時に明度-10%
- カード: 丸角12px、border 1px #E5E7EB
```

### 2. 参考デザインの用意

- [Dribbble](https://dribbble.com/)や[Pinterest](https://pinterest.com/)で似たデザインを探す
- スクリーンショットを保存
- 「このスタイルに従って」とAIに指示

### 3. ロゴ・ブランド素材

- 既存のロゴデータ（SVG推奨）
- ブランドカラーのHEXコード
- 参考にしたい既存サイトのURL

:::message alert
**注意**: 著作権のある画像をそのまま使うのはNG。「スタイルの参考」として使いましょう
:::

# Step 3: Orchestration（構築・指揮）

ここからようやくAntigravityの出番です！

## Planning Agentの使い方

1. **プロジェクト作成**
   - Antigravityで新規プロジェクトを開始
   - Step 1で作ったSOP（仕様書）を貼り付け
   - Step 2のデザインファイルをアップロード

2. **実装プランの確認**
   - AIが自動で実装計画を生成します
   - タスクリスト形式で表示されます

### アノテーション機能（超重要！）

実装プランの**特定の行**にコメントを追加できます:

```
例:
AIの提案: 「ヘッダーコンポーネントを作成」
        ↓
あなたの追加指示: 「ヘッダーは固定表示（sticky）にして、
                  スクロール時に影を追加してください」
```

**なぜ重要?**

- 全体を書き直さなくても、ピンポイントで修正できる
- AIが「どこを」「どう直すか」を正確に理解できる

## Agent Managerで並列作業

![Agent Managerの開き方](https://github.com/Inosuke0718/Zenn/blob/main/images/open-agent-manager.png?raw=true)

複数のAIを**同時に**働かせます:

### 使用例

**メインエージェント**: アプリのコード実装
**サブエージェント1**: 競合アプリのリサーチ → `research.md` に出力
**サブエージェント2**: テストケースの作成 → `test-cases.md` に出力

これらは「ユニバーサル・インボックス」で一元管理できます。

# Step 4: Workflows（自動化・保守）

「コードは書けた。でも、テストやデプロイ、バグ修正は誰がやるの？」

FLOWフレームワークの最終段階、**Step 4: Workflows**は、まさにこの問いに対する答えです。ここでの主役は、あなたではなく **自律的に動くAIエージェント** です。

Step 1〜3で作ったアプリケーションを、AI自身に「守らせ（保守）」、「育てさせる（運用）」ための仕組み作り。それがStep 4の本質です。


## 1. Self-Healing Tests（自己修復テスト）

Step 4の目玉機能であり、従来の開発と決定的に違うのがこの **「自己修復（Self-Healing）」** です。

Google Antigravityの **Browser Agent** を使用し、AIに以下のような指示を与えます。

> 「ユーザーとしてログインし、商品をカートに入れ、決済画面まで進んでエラーが出ないか確認して」

AIはブラウザを立ち上げて操作を行いますが、もしここでエラーが発生したらどうなるでしょうか？

### 従来のテスト vs FLOW Step 4

| 従来のテスト                                 | FLOW Step 4                                              |
| :------------------------------------------- | :------------------------------------------------------- |
| テストが落ち、赤いログが出て、人間が呼ばれる | AIがエラーログ（コンソールやネットワークタブ）を読み取る |
| 人間がデバッグし、原因を特定                 | 「決済ボタンのIDが変わっている」などの原因を特定         |
| 人間がコードを修正                           | **AIがソースコードを勝手に修正する**                     |
| 人間が再テストを実行                         | もう一度テストを実行し、成功することを確認               |

**人間が寝ている間に、AIがバグを見つけ、勝手に直して報告だけ上げてくれる。** これがStep 4が目指す「Workflows」の姿です。

### 実際の動き

```
シナリオ: ECサイトの決済フローテスト

1. AIがブラウザを起動
2. ユーザーとしてログイン
3. 商品をカートに追加
4. 決済画面に遷移 → エラー発生！
5. AIがコンソールログを解析 → 「決済ボタンのIDが変更されている」と特定
6. AI自身がコードを修正
7. 再テスト → 成功 ✅
8. 修正内容をレポートとして出力
```

## 2. MCPによるDevOpsの完全自動化

コーディング以外の「面倒な作業」もすべてAIに委任します。ここで鍵となるのが **MCP (Model Context Protocol)** です。

AIを単なるチャットボットから、GitHubやVercelを操作できる「DevOpsエンジニア」へと進化させます。

### 具体的なワークフロー例

自然言語でこう頼むだけで、裏側で複雑なパイプラインが走ります。

> 「今の変更をGitHubにプッシュして、本番環境（Vercel）にデプロイして。終わったらURL教えて」

AIはMCPを通じて以下のツールを直接操作します:

- **GitHub**: コミットメッセージの自動生成、プッシュ、プルリク作成
- **Vercel / Netlify**: ビルド、デプロイ、プレビューURLの発行
- **Slack / Discord**: 「デプロイ完了しました（URL: xxx）」とチームへの通知

### 対応サービス例

| カテゴリ         | サービス         | 用途                             |
| :--------------- | :--------------- | :------------------------------- |
| ソースコード管理 | GitHub           | コード変更の自動プッシュ、PR作成 |
| データベース     | Supabase         | スキーマの同期、マイグレーション |
| デプロイ         | Vercel / Netlify | 自動デプロイ、プレビューURL発行  |
| 通知             | Slack / Discord  | 進捗通知の送信                   |

```
設定例:
1. AntigravityでMCPサーバーを追加
2. GitHubのトークンを入力
3. 「変更をmainブランチにプッシュして」と指示するだけ
```

## まとめ: あなたは「開発者」から「管理者」になる

Step 4: Workflowsを構築することで、開発者の役割は大きく変わります。

| Before（従来の開発者） | After（Step 4導入後）            |
| :--------------------- | :------------------------------- |
| コードを書く           | 「テストしておいて」と指示を出す |
| テストを書く           | AIが自律的に動くのを監視する     |
| デプロイコマンドを叩く | 「デプロイして」と指示を出す     |

アプリを作って終わりではなく、**「アプリが勝手に成長・維持されるシステム」** を構築するのが、FLOWフレームワークのゴールです。

# 実践: 簡単なToDoアプリを作ってみよう

## 準備（10分）

### 1. Frameフェーズ

Claudeとの対話:

```
あなた: 「初心者向けのシンプルなToDoアプリの仕様を、
        500文字以内で作ってください。
        技術スタックはPython（Streamlitなど）で」

Claude: （SOPを生成）
```

### 2. Layoutフェーズ

`design.md` を作成:

```markdown
# ToDoアプリ デザインシステム

カラー:

- Primary: #6366F1（インディゴ）
- Success: #10B981（グリーン）
- Text: #1F2937

フォント: Noto Sans JP
ボタン: 丸角6px、影なし
```

参考画像: Dribbbleで「minimalist todo app」を検索してダウンロード

## 実装（20分）

### 1. Antigravityで新規プロジェクト作成

### 2. 準備したファイルをアップロード

- SOP（仕様書）
- design.md
- 参考画像

### 3. プロンプト入力

```
この仕様書とデザインガイドに従って、
ToDoアプリを実装してください。
参考画像のスタイルを踏襲してください。
```

### 4. 実装プランを確認

AIが提案したタスクリストをチェック:

```
☐ プロジェクトのセットアップ
☐ ToDoリスト表示機能の実装
☐ 追加フォームの実装
☐ 削除機能の追加
☐ データの永続化（SQLite/CSVなど）
☐ レスポンシブ対応
```

### 5. アノテーションで微調整

```
タスク「ToDoリスト表示機能の実装」に追加:
→ 「完了済みタスクには取り消し線とグレーアウトを適用」
```

### 6. 実装開始！

Planning Agentが自動でコードを生成します。

### 7. Browser Agentでテスト

自動的にブラウザが立ち上がり、動作確認 → バグがあれば自動修正

# よくある失敗パターンと対策

## ❌ 失敗例1: いきなりコードを書かせる

```
「ToDoアプリ作って」
→ 曖昧すぎて、何度も手戻りが発生
```

**対策**: Frame（構想）を必ず作る

## ❌ 失敗例2: デザインを口頭で説明

```
「青っぽい感じで、モダンな雰囲気にして」
→ AIの解釈がバラバラで統一感がない
```

**対策**: デザインファイルと参考画像を用意

## ❌ 失敗例3: すべて1つのエージェントにやらせる

```
「コード書いて、テストして、ドキュメントも作って」
→ 処理が遅く、途中で止まることも
```

**対策**: Agent Managerで作業を分散

# まとめ: FLOW思考で開発効率10倍に

## FLOWの本質

AIを「便利なツール」ではなく、**「優秀なチームメンバー」**として扱うこと。

### 従来の開発

```
あなた → AIに1行ずつ指示 → 手戻り → また指示 → 手戻り...
```

### FLOW型開発

```
あなた（マネージャー） → 明確な設計書を渡す
                    → 複数のAIが並行作業
                    → 自動テスト・修正
                    → 完成！
```

## 次のステップ

1. **小さいプロジェクトで試す**: まずは簡単なToDoアプリや計算機から
2. **テンプレートを作る**: よく使うSOPやデザインファイルをストック
3. **コミュニティに参加**: [Antigravity Discord](https://discord.gg/antigravity)などで情報交換
