# 【Gitでは頼むからこれだけは知っていて】初心者でも迷わない！Git & GitHub の基本を優しく解説 🌲

「Gitって、黒い画面で呪文をたたくやつでしょ...？」
「GitHubとGitの違いもよくわらかないし、壊しそうで怖い...」

そんな Gitアレルギーを持っている初心者の方、いませんか？実は、最初に覚えるべきことはそれほど多くありません。

この記事では、 たのむから「これだけは知っておいてほしい！」 というGitとGitHubのエッセンスをぎゅっと凝縮して解説します。

---

## 🌳 Gitとは

Gitは、ソースコードやファイルの変更履歴を記録・管理するための **「分散型バージョン管理システム」** です。

一言でいうと「いつ・誰が・どこを変更したか」を積み上げて残し、必要なら過去の状態に戻したり、複数人の変更を安全に統合したりできる仕組みです。

### 📥 Gitのインストール

Gitを使い始めるには、まずPCにインストールする必要があります。公式サイトから自分のOSに合ったインストーラーをダウンロードしてください。

[Git 公式ダウンロードページ](https://git-scm.com/downloads)

## ☁️ GitHubとは

GitHubは、Gitで管理しているリポジトリ（変更履歴込みのデータ）をインターネット上でホスティングし、共有・コラボしやすくする **「Webサービス」** です。

チーム開発での共有だけでなく、個人のポートフォリオ公開や、他人のOSS（オープンソース）への参加にもよく使われます。

### 🔑 GitHubのアカウント作成方法

GitHubは公式サイトにアクセスし、トップページの「Sign up」から登録を開始します。

1. [GitHub公式サイト](https://github.com/) にアクセスします。
2. 登録では、ユーザー名（username）・メールアドレス（email）・パスワード（password）などを入力して進めます。
3. 登録後は、必要に応じてメール認証などを行い、ログインできればアカウント作成は完了です。

画面の案内に従って進めればOKです。ここからエンジニアとしての道が本格的に始まります。

## 🕵️‍♂️ GitとGitHubの違い

Gitは「ローカルPCでも動くバージョン管理のツール（仕組み）」で、GitHubは「Gitを使ったリポジトリを置いて共同作業しやすくするWebサービス」です。

| 項目     | Git                  | GitHub                      |
| :------- | :------------------- | :-------------------------- |
| **役割** | 管理ツール（仕組み） | Webサービス（ホスティング） |
| **環境** | ローカルPC           | インターネット上            |

たとえば、Gitだけでも履歴管理はできますが、GitHubがあると“オンライン上で共有する・レビューする・共同で進める”がやりやすくなります。

## 📖 最初に覚えるべき「Gitの魔法の言葉」8選

Gitを使い始めると呪文のような単語が出てきます。まずはこの8つだけ頭の片隅に置いておきましょう。

### 1. リポジトリ（Repository）🗃️

変更履歴をまるごと保存する **「箱」** のことです。

- 自分のPCにあるのが： **ローカルリポジトリ**
- GitHub上にあるのが： **リモートリポジトリ**

### 2. アド（Add）➕

変更したファイルをコミット対象として **「準備（ステージング）する」** 操作です。

### 3. コミット（Commit）💾

変更内容をリポジトリに **「記録する」** 操作です。ゲームでいう「セーブポイント」を作るイメージですね。

### 4. ブランチ（Branch）🌿

開発の作業ラインを **「枝分かれ」** させる機能です。メインのコードを壊さずに、新しい機能を試したい時に使います。

### 5. マージ（Merge）🤝

分岐したブランチを、別のブランチに **「合流」** させる操作です。新機能が完成したら、メインの枝に合体させます。

### 6. クローン（Clone）👯

GitHub上にある「リモートリポジトリ」を、自分のPCにまるごと **「コピー」** する操作です。

### 7. プッシュ（Push）📤

自分のPCで記録したコミットを、GitHub（リモート）へ **「アップロード」** することです。

### 8. プル（Pull）📥

GitHub上の最新の変更を、自分のPCへ **「ダウンロード」** して反映させることです。

## 🔄 じゃあ実際にどう使うの？ 〜 8つの言葉が登場する流れ

上の8つの言葉を使うと、実際の開発はこんな流れになります。  
**「GitHubにあるプロジェクトに参加して、新機能を追加してみる」** というシナリオで見てみましょう。

![alt text](https://github.com/Inosuke0718/Zenn/blob/main/images/git-flow.png?raw=true)

---

### STEP 1：まず「クローン」でPCにコピーする 👯

GitHubに置いてある**リモートリポジトリ**を、自分のPCに持ってきます。

```bash
git clone https://github.com/ユーザー名/リポジトリ名.git
```

> 📌 これで自分のPCに「ローカルリポジトリ」が作られました。

---

### STEP 2：「ブランチ」を作って作業する 🌿

メインのコードを壊さないよう、作業用の枝を分けます。

```bash
git branch new-feature     # ブランチを作る
git checkout new-feature   # そのブランチに移動する
```

> 📌 これで「main」には影響せず、安心して作業できます。

---

### STEP 3：ファイルを編集したら「アド」する ➕

ファイルを変更し終えたら、コミット対象として準備（ステージング）します。

```bash
git add .
```

> 📌 「どのファイルの変更を記録するか」を選ぶイメージです。

---

### STEP 4：「コミット」でセーブポイントを打つ 💾

ステージングした内容を、メッセージとともに記録します。

```bash
git commit -m "新機能のヘッダーを追加"
```

> 📌 コミットメッセージは「何をしたか」が一言でわかるように書くのが慣習です。

---

### STEP 5：「プッシュ」でGitHubにアップロードする 📤

ローカルのコミットをリモートリポジトリ（GitHub）に送ります。

```bash
git push origin new-feature
```

> 📌 これでチームメンバーもあなたの変更を見られるようになります。

---

### STEP 6：「マージ」でメインに合流させる 🤝

新機能が完成したら、`main` ブランチに合体させます。  
GitHubでは「Pull Request（プルリクエスト）」という形でレビューを挟んでからマージするのが一般的です。

```bash
git checkout main
git merge new-feature
```

---

### STEP 7：「プル」で最新状態を取り込む 📥

チームで開発していると、自分が作業していない間に誰かが変更を加えている場合があります。作業開始前にプルして最新の状態を取り込みましょう。

```bash
git pull origin main
```

---

### 🗺️ 全体の流れを図で整理

```
GitHub（リモート）
    ↓ clone（最初の1回だけ）
自分のPC（ローカル）
    ↓ branch（作業用ブランチを作る）
    ↓ add（変更をステージング）
    ↓ commit（変更をセーブ）
    ↓ push（GitHubにアップロード）
GitHub（リモート）
    ↓ merge（mainに合流）
自分のPC（ローカル）
    ↓ pull（最新を取り込む）
```

> 💡 **まとめると**：「clone → branch → add → commit → push → merge → pull」がGitを使った開発の基本サイクルです。この流れを体で覚えれば、チーム開発にもスムーズに入っていけます！
